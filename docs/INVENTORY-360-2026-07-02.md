# INVENTORY-360 — inventory + procurement as a daily-use tool (2026-07-02)

*Read-only focused review against main (code = `6184961`). Question: can the storekeeper, the buyer, and the owner-verifier actually run the store in this? Companion practice research feeds `SPEC-0023-stock-take-jard.md`. Paths relative to `apps/farm-os/`.*

## Grades

| Persona | Grade | Verdict |
|---|---|---|
| Storekeeper | **D+** | Receives against PRs beautifully (partials, over-receipt guard, Arabic errors) — and can do almost nothing else: no item create/edit, no wastage/adjustment, no transfer, no count. |
| Buyer | **C−** | Coverage→PR→approve→receive works end-to-end; but no open-orders console, supplier frozen from `preferred_supplier_id`, zero price intelligence, engine's stale-PO drop invisible. |
| Owner-verifier | **D** | Foundation A-grade (append-only ledger, DB audit) — surface near-F: anonymous movements, 12-row cap, no export, no valuation, no shrinkage view, no جرد. He must trust the storekeeper because the UI gives him nothing to check. |

**Strategic line:** anti-leakage is enforced only on the digital ledger; physical stock has no reconciliation loop, so leakage moves to the physical/book boundary — exactly where the paper forgery mode lives.

## Top 10 gaps (anti-leakage × daily-use; each one PR)

1. **No stock-take (جرد) workflow at all** — zero hits repo-wide; every primitive exists (movement types `adjustment`/`loss` in `20260622000005:49-50`, internal `fn_post_movement`, the PR SoD pattern). → **SPEC-0023** Slice 1.
2. **Movements are anonymous on screen**: no `created_by` on `inventory_movements` (`0005:45-61`; `fn_post_movement` inserts none); `audit_log` captures the actor but **no page anywhere reads audit_log**. An investigation can't name a person from any screen. → add `created_by` (backfill from audit_log) + a «بواسطة» column on the item ledger.
3. **No ad-hoc issue/adjustment/write-off** — only receipt + execution can move stock; spoilage/spillage/expiry physically leaves but the book never decrements → phantom stock that hides theft inside honest drift. → `fn_post_adjustment` wrapper (mandatory reason, loss/adjustment/expiry only, flagged for owner review) + small form.
4. **Actual price paid never captured at receipt**: all three `fn_post_receipt` generations post `null -- p_unit_cost` (`0024:93`, `0045:263`, `20260701210000:112`); no invoice_no; ReceiveForm captures qty only. "Who sold us potassium cheapest last 3 times" is unanswerable; invoice-padding invisible. → `p_unit_costs jsonb` + `p_invoice_no` + form fields; price history falls out free.
5. **Item master 100% read-only**: zero insert/update/delete on `inventory_items` in the whole app (all `.select()`); items exist only via `supabase/seed.sql:184-189`; no import descriptor; no archive. Unit is free text nobody can set (the UI half of #216); `pack_size`/`unit_cost`/reorder-policy written nowhere in-app. RLS write-gates ready (`0066`). → item CRUD form (unit from a small vocabulary, category, pack_size, unit_cost, reorder policy, archive).
6. **Ledger view: last 12 rows, no export, no filters** (`[itemId]/page.tsx:45-50,228-244`). Owner can't pull "all losses this month". → `/inventory/movements` FilterableTable page (type/location/date + CSV).
7. **No stock valuation anywhere** (no on_hand × unit_cost in the codebase). → value column + total (labeled «تقديرية» per #89-B honesty) + one dashboard KPI.
8. **No open-orders console; engine's stale-PO drop invisible**: PR list has no status/overdue filters; engine silently stops counting POs with `needed_by < v_period_start` (`0034:141`, re-emitted `20260701180000:421`) while the detail-page banner uses a different rule. → filter chips + a «لم يعد محسوباً في التغطية» badge computed with the engine's own rule.
9. **Expiry modeled, never captured, never alerted**: `expiry_date`/`batch_no` exist on movements, `expiry_tracked` on items, `fn_bin_rebuild` nets expiry — ReceiveForm captures neither; no dashboard card. "It expired" is the classic cover story, currently unverifiable. → expiry/batch inputs for tracked items + «ينتهي قريباً» card.
10. **Receiving is single-person, one click, evidence-free**: no invoice ref, no delivery-note photo (`attachments.entity_type` CHECK is structure-only — `20260622000082:18`), the «استلام كل المتبقي» ghost button posts a whole PR in one tap by the gatekeeper himself. → extend attachments to `purchase_request` + required delivery-note photo (or explicit «بدون مستند» flag).

Noted below top-10: `rejected` PR status has no UI trigger; supplier master lacks contact-person/active flag; supplier-360 truncates (20 purchases/12 expenses, no AP balance); no transfer UI despite the transfer-guard migration.

## Protect (do not weaken while building)
