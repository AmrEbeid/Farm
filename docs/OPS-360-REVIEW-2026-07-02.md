# OPS-360-REVIEW — operations modules as a daily-use product (2026-07-02)

*Read-only focused review of the planning workspace + operations + execution + activity modules against main (code = `6184961`). Distinct question from `REVIEW-360-2026-07-01.md` (code quality): **can a farm manager actually plan and run a week in this?** Feeds `OPS-PLAN-2026-07.md`. Paths relative to `apps/farm-os/`.*

## Grade: C− as a daily-use planning tool (code quality separately: A−/B+)

The system is an excellent **operation recorder** and a weak **work planner**. The create path is genuinely strong (one atomic drawer: N materials incl. spray-compliance fields, N labor lines, multi-day, assignees+lead, harvest stage, irrigation basis, relative dependency — `fn_add_plan_operation_multi`, 16 args). The execute path (`/m`) is real phone-first craft. But after creation an operation is **effectively immutable and monostate**, whole-farm-scoped, and invisible on any calendar.

## The actual state machines

**plans** (`fn_set_plan_status`, 0084): `draft → active → closed` + `active → abandoned` — but the RPC validates the *value not the transition* (any jump accepted via PostgREST), there is no reopen, and **plan status gates nothing** (`/m` selects ops with no plan-status filter; `fn_execute_operation` never reads `plans.status`; closing a plan leaves its ops executable, no warning).

**plan_operations** (9 CHECK statuses, 0058): the app can write exactly ONE transition —
```
planned ──fn_execute_operation──► done        (claim-first; the only writer, verified by grep)
   ├─ [NO PATH] approved / reserved / ready / in_progress / blocked / abandoned / skipped
   ├─ [NO PATH] reschedule (nothing ever UPDATEs planned_at; only depends_on_* is updatable)
   └─ done: terminal — [NO PATH] reopen/correct
```
The guards for blocked/abandoned/skipped exist and are correct (0057) — **defensive code for states the product cannot produce** (reachable only by owner/farm_manager raw PostgREST). `approval_needed` is hardcoded `true` and rendered as a permanent "موافقة؟ نعم" decoration. Agronomist sign-off exists but gates nothing (documented in `plans/[planId]/actions.ts:544-556`).

**farm_event** (ad-hoc): planned|in_progress|done at create + `fn_set_event_status` → done. Ironically the *ad-hoc* path has more lifecycle than the *planned* path.

## Top 12 gaps ranked by daily-use pain (evidence + one-PR fix each)

1. **Hawsha-level planning unreachable.** `fn_create_plan` supports farm|sector|hawsha scope, but `PlanCreateForm.tsx` renders only type+dates and `plans-actions.ts:31` hardcodes `p_scope_type:"farm", p_scope_id:null`; ops inherit the null target → the plans/dashboard cost-by-scope and cost-per-palm charts are structurally empty, and the `20260701390000` execute-rollup has nothing to resolve. "Fertilize hawsha 3 Tuesday" cannot be entered. → scope picker + per-op target picker (RPC already stores the columns).
2. **No reschedule.** Rained-out op = execute on the wrong day or rot as "متأخرة". → `fn_reschedule_plan_operation(op,date,reason)` + audit + date-edit UI.
3. **No cancel/block from UI.** Labels (`OP_STATUS_AR`) + guards exist, zero setters. → `fn_set_plan_operation_status` (mirror `fn_set_event_status`) + buttons on `/m/execute/[opId]` + plan table.
4. **A real week ≈ 150–200 serial interactions.** Per op via `OperationBuilder.tsx` ≈ 10–15 inputs, one drawer at a time; no duplicate-op, no copy-last-week, no bulk assign. → duplicate-operation action (re-call the multi RPC with a new date).
5. **No calendar/week view anywhere** (grep: no calendar component; `/m`'s overdue/today/upcoming buckets are the closest thing to a schedule in the product). → read-only 7-column week grid on the plan page.
6. **Templates invisible in prod.** `plan_operation_templates` + instantiate RPC are solid, but the 3 programs live only in `supabase/seed.sql:240-275` (dev), there's no authoring UI (explicit non-goal, `OperationTemplatePicker.tsx:18-20`), and the picker hides itself when `templates.length===0` (line 69). Instantiate also hardcodes `ends_on=null` and ignores scope/palm counts (delta to SPEC-0021). → template CRUD + ship programs as prod data.
7. **Dedup silently swallows same-day ops.** Find-or-create on `(plan_id, subtype, planned_at)`; a second same-day irrigation (the product even has `preferred_time_of_day`!) returns `deduped:true` — and `actions.ts:395-399` discards the flag and reports success. → client idempotency key or loud dedup surfacing.
8. **Backdating impossible.** `fn_execute_operation` hardcodes `occurred_at = v_now` (all re-emits) — night-time recording shifts PvA dates and REI/PHI windows (`computeSprayComplianceWindow` reads `occurred_at`). `fn_record_event` accepts `p_occurred_at` but `RecordActivity.tsx` renders no date field. → bounded optional backdate + date inputs.
9. **No partial completion/carry-over.** `ends_on` models the span but execution is a single flip to done — day 1 of a 3-day pruning op either shows overdue or closes with under-consumed materials. → progress events linked via `data->op_id` + explicit complete; or minimally an in_progress toggle (#3's RPC).
10. **Assignment one-way, workload invisible.** Assignees set only at create; later only removable (`OperationAssignees.tsx` — no add-picker); supervisor can't assign (dead `fn_assign_plan_operation`); person 360 never joins `plan_labor_requirements` — "who is free Thursday?" unanswerable; crews are free text. → assign RPC + add-picker + person-week grouping.
11. **Ad-hoc work is second-class in every rollup.** `RecordActivity` is pleasantly easy and rolls up ancestors correctly, but free-text subtype never matches the vocabulary, no material issue, no cost, `plan_id=null` → invisible to PvA and actual-cost joins. Unplanned work — half a real farm's day — accumulates as unpriced notes. → subtype select (free-text fallback) + optional material line via `fn_post_movement`.
12. **The manager's Monday questions unanswerable:** what slipped and why (no reschedule/reason history), crew utilization (labor_logs hours never linked to ops), cost per operation *type* this season (dashboard has no subtype dimension, `dashboard/page.tsx:124-197`), farm-wide awaiting-sign-off queue (per-plan card only). → subtype chart + global sign-off queue (pure reads).

Honorable mentions: weather check permanently "warn — لم يُقيَّم" while a weather module exists (`actions.ts:139`); `runPlanChecks` delete-then-insert can leave plan_checks empty mid-failure (acknowledged comment, `actions.ts:150`); the REI/PHI banner is only seen by someone revisiting the op *after* completion — the next crew never navigates there.

## What already delights — protect

- `fn_add_plan_operation_multi` as one atomic authoring transaction (+ the exemplary 5-PR layered re-emit discipline).
- The `/m` field surface: buckets, "مهامي فقط" honoring both lead + assignees, non-executable affordances hidden, prefill from planned values never magic defaults, no stranded spinners, per-requirement actual keys.
- The operation vocabulary (15 real date-palm subtypes with seasonal ordering, حصاد stages, spray target zones, `preferred_time_of_day`, soil-test irrigation basis) — **the domain modeling is ahead of the workflow**.
- Relative scheduling done safely (metadata + read-time computation, no planned_at-rewrite cascade) with live effective-date preview.
- Honest-state conventions (weather never fakes green; deduped counts in the template picker; sign-off refuses silent re-stamp).
- `fn_record_event`'s ancestor-chain rollup — the backbone the fixes should reuse.

**Bottom line:** the shortest path C− → B+ is gaps #1 + #2/#3 + #4 — three small PRs that let a manager actually run a week, and which SPEC-0021 silently depends on (per-hawsha proposal targets need #1's plumbing).
