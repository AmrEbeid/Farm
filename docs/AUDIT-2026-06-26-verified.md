### A1 — Stale/overdue approved PO projected as guaranteed period-1 supply → **masks a real shortage** · **HIGH** · GATED (stock engine)
- **Where:** `supabase/migrations/20260622000034_engine_stale_po_guard.sql:131-146` (receipts query filters only `pr.needed_by >= v_period_start`, line 141; `v_period_start = min(planned_at)`, line 88). - **Cardinal sin:** an undelivered approved PO masks a shortage *forever* (`fn_post_receipt` only flips approved→received on actual receipt, `0024:70`). Migration header lines 20-22 already document this as a deferred residual of #197.
- **Interaction (A2 below):** the app PR-create path hardcodes `needed_by="2025-07-08"`, manufacturing exactly the stale PO this path consumes.

- **Minimal fix:** in the receipts query add `and pr.needed_by >= greatest(v_period_start, current_date)` (or exclude `needed_by < current_date`). New `create or replace` migration. Add a regression asserting shortage stays true for `needed_by < current_date`.

### A11 — pgTAP CI harness is blind to test-file errors & plan-count mismatches (**false-green**) · **HIGH** · GATED (CI/deploy gate)
- **Where:** `supabase/test-shims/run-pgtap-local.sh:36-49` (sole gate per `.github/workflows/db-tests.yml:46`). psql runs inside `out="$(psql … 2>&1)"` (`:37`) so its exit code is discarded; `set -e` does not abort. Pass/fail counted only by `grep '^ok'` / `^not ok` (`:38,42`); verdict `tot_no -eq 0` (`:49`). **Empirically reproduced** (PG 17.10 + pgTAP 1.3.5): a test that errors before any assertion → 0 ok / 0 not-ok → run exits 0; a mid-test abort → plan(5)-vs-ran-2 mismatch invisible. Migrations are safe (`ON_ERROR_STOP=1`, `:30`); only the test loop is blind — exactly the schema-drift class a DB gate must catch.
- **Why GATED:** this is the gate the Owner relies on for every engine/RLS migration below — it can certify green while tests silently never run.
- **Fix:** assert psql exit status (`$?`/`PIPESTATUS`); treat `ERROR:`/`FATAL:`/`Looks like you planned` as failures; optionally parse `plan(N)` and assert `ok+not_ok == N`. (`pg_prove` already does this.)
