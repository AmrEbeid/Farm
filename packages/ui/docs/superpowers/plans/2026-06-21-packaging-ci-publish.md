# Packaging, CI & Private Publish Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire `@amrebeid/ui` for versioned, gated, private publishing — Changesets for versioning + CHANGELOG, GitHub Actions CI that red-gates merges, GitHub Packages (scope `@farm-os`) as the private registry with a documented npm-private alternative, a `prepublishOnly` build/typecheck/test gate, and a verified consumer-facing package surface (`.` + `./styles.css`, ESM+CJS+d.ts, `sideEffects`). Stay `0.x` and `private:true` until the final task cuts the publish-ready 1.0.

**Architecture:** This phase is **infrastructure, not React components**, so TDD is adapted: each task writes real config files and is "tested" by running a concrete verification command with an expected result (not a Vitest unit test). The pieces layer up: Changesets gives versioning + CHANGELOG; two GitHub Actions workflows (`ci.yml` runs the full quality gate on PR/push; `release.yml` versions + publishes on `main` via `changesets/action`); `.npmrc` + `publishConfig` point publishing at GitHub Packages; `prepublishOnly` re-runs the gate locally before any `npm publish`; `npm pack --dry-run` is the oracle that the published tarball contains only `dist/` (incl. `styles.css`). The final task flips `private` off, sets `version` to `1.0.0`, and documents cutting 1.0.

**Tech Stack:** npm, Node 20 (`.nvmrc`), @changesets/cli, GitHub Actions, GitHub Packages (npm registry), tsup (existing build), Vitest + token-purity/token-presence scripts (existing gates from the theming-foundation phase).

## Global Constraints
- **Stay `0.x` until 1.0.** The API is still moving; only the final task (Task 8) cuts `1.0.0` as the publish-ready milestone. Until then `version` stays `0.x` and Changesets bumps stay in the `0.x` range.
- **Private-first.** `"private": true` stays in `package.json` until Task 8. Nothing in CI or the registry config requires going public; the setup is *public-capable* but ships privately to GitHub Packages for sub-project B.
- Publish target is the **`@farm-os` scope on GitHub Packages**; the npm-private alternative is documented (`CONTRIBUTING.md`) but not the default.
- Do **not** modify source code (`src/**`) or component CSS in this phase — packaging/CI/docs files only.
- Keep the existing public package surface intact: `exports` (`.` + `./styles.css`), ESM (`.js`) + CJS (`.cjs`) + `.d.ts`, `"sideEffects": ["*.css"]`, `"files": ["dist"]`. Task 7 verifies it; it does not redesign it.
- Match the repo's existing JSON/script style (2-space indent, double quotes, scripts named in the existing `tokens:*` / `build:*` colon style).
- Node version is pinned by `.nvmrc` (`20`); CI must read Node from `.nvmrc`, never hardcode a different major.
- Commit after every task: `git add` the **exact** paths listed, then `git commit -m "<type>(<scope>): ..."` with a body ending:
  `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`
- The repo currently has **no git remote** and no `origin`. The workflow YAML and registry config are authored to be correct for `github.com/farm-os/farm-os-ui` (scope `@farm-os`); the operator wires the actual remote + repo secrets out-of-band (noted in Task 8).

**Interfaces:**

- [ ] **Step 1: Create `.github/workflows/ci.yml`**

```yaml
name: ci

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true

jobs:
  build:
    name: typecheck · tokens · test · build · storybook
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node from .nvmrc
        uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc

      - name: Typecheck

      - name: Token presence
        run: npm run tokens:present

      - name: Token purity
        run: npm run tokens:purity

      - name: Build (ESM + CJS + d.ts + styles.css)
        run: npm run build

      - name: Build Storybook
        run: npm run build-storybook

Notes:
- `node-version-file: .nvmrc` reads Node `20` — never hardcode a major.

### Task 6: Release workflow (`release.yml`)

**Files:**
- Create: `.github/workflows/release.yml`

**Interfaces:**
- Consumes: the `version` + `release` scripts (Task 3), the `.npmrc` registry routing (Task 4).
- Produces: a `release` workflow on push to `main` that uses `changesets/action` to either open a "Version Packages" PR (when changesets are pending) or run `npm run release` to publish to GitHub Packages (when the version PR is merged). Gated behind `private:false` (Task 8) — until then `changeset publish` is a no-op for a private package, which is the intended pre-1.0 behavior.

- [ ] **Step 1: Create `.github/workflows/release.yml`**

```yaml
name: release

on:
  push:
    branches: [main]

concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false

permissions:
  contents: write
  pull-requests: write
  packages: write

jobs:
  release:
    name: version or publish (changesets)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node from .nvmrc
        uses: actions/setup-node@v4
        with:
          node-version-file: .nvmrc
          cache: npm
          registry-url: https://npm.pkg.github.com
          scope: "@farm-os"

      - name: Install
        run: npm ci

      - name: Create release PR or publish
        uses: changesets/action@v1
        with:
          version: npm run version
          publish: npm run release
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Notes:
- `setup-node` with `registry-url` + `scope` writes a CI `.npmrc` and exports the token as `NODE_AUTH_TOKEN`, matching the committed `.npmrc` placeholder.
- `permissions: packages: write` is what lets the built-in `GITHUB_TOKEN` publish to GitHub Packages for the `@farm-os` org — no extra PAT needed for same-org publish.
- `version: npm run version` runs `changeset version`; `publish: npm run release` runs `changeset publish`. While `private:true`, `changeset publish` skips the package — safe pre-1.0.

- [ ] **Step 2: Validate the YAML wires the gate correctly**

Run:
```bash

```
Expected: `release.yml ok`. (If `actionlint` is installed: `actionlint .github/workflows/release.yml` → exit 0.)

```markdown
# Contributing to @amrebeid/ui

## Adding a changeset (required for any user-facing change)

Every PR that changes the published surface (components, props, tokens, styles,
exports) must include a changeset. From the repo root:

```bash
npm run changeset
```

Pick the bump type and write a one-line summary:

- **patch** — bug fix, token tweak, internal change with no API change.
- **minor** — new component, new prop, additive theming dimension (the norm while we are `0.x`).
- **major** — breaking API change. **Do not select major before 1.0** — while in `0.x` use minor for breaking changes (semver pre-1.0 convention) unless the change is the deliberate 1.0 cut.

Commit the generated `.changeset/*.md` file alongside your code. CI does not
require it mechanically, but a PR without a changeset will not bump the version
or appear in the CHANGELOG.

## How releases happen

On merge to `main`, the `release` workflow runs `changesets/action`:

1. If unreleased changesets exist, it opens/updates a **"Version Packages"** PR
   that applies the bumps and regenerates `CHANGELOG.md`.
2. When that PR is merged, the workflow runs `npm run release`
   (`changeset publish`) and publishes to **GitHub Packages** under the
   `@farm-os` scope.

While the package is `"private": true` (pre-1.0), `changeset publish` is a no-op
— version PRs and the CHANGELOG still work; nothing is published yet. This is
intentional: we ship to consumers privately only from 1.0 (see below).

## Registry

Default registry is **GitHub Packages** (`https://npm.pkg.github.com`), routed by
`.npmrc` (`@farm-os:registry=...`) and enforced by `publishConfig` in
`package.json`. Same-org publishing uses the built-in `GITHUB_TOKEN` (no PAT).

### Installing as a consumer

Consumers add an `.npmrc` with the scope route and a read token:

```ini
@farm-os:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${NODE_AUTH_TOKEN}
```

then `npm i @amrebeid/ui` with `NODE_AUTH_TOKEN` set to a `read:packages` token.

### Alternative: npm private registry

To publish to npm's private registry instead of GitHub Packages:

1. In `package.json` set `"publishConfig": { "registry": "https://registry.npmjs.org", "access": "restricted" }`.
2. Replace `.npmrc` with `//registry.npmjs.org/:_authToken=${NODE_AUTH_TOKEN}` and drop the `@farm-os:registry` GitHub line.
3. In `release.yml` set `registry-url: https://registry.npmjs.org` and provide `NODE_AUTH_TOKEN` from an `NPM_TOKEN` secret (an automation token with publish rights).

Both paths satisfy the publish-ready gate; GitHub Packages is the default because
the repo already lives in the `@farm-os` GitHub org.

## Cutting 1.0

1.0 is the publish-ready milestone (full catalog + theming + green CI). To cut it:

1. Land a changeset that bumps to `1.0.0`.
2. Set `"private": false` in `package.json`.
3. Merge the "Version Packages" PR — `release.yml` then publishes `@amrebeid/ui@1.0.0`
   to GitHub Packages via `prepublishOnly` (typecheck + test + build) gate.
```
